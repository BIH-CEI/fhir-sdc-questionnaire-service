0. Environment Setup

  • Task 0.1: Start the workshop environment

    Clone the repository and start all services.

    git clone https://github.com/nicktiberi/fhir-sdc-questionnaire-service.git
cd fhir-sdc-questionnaire-service
docker compose -f docker-compose.workshop.yml up -d

  • Task 0.2: Verify services are running

    Check that all services are healthy.

    # Check service status
docker compose -f docker-compose.workshop.yml ps

# View logs (wait for "Started Application")
docker compose -f docker-compose.workshop.yml logs -f

  • Task 0.3: Access the Demo UI

    Open http://localhost:8000 in your browser.

1. Basic PROMIS Exchange Working

Demonstrate that standard FHIR questionnaire exchange works correctly.

  • Task 1.1: Retrieve a Questionnaire from Site A

    Get the PROMIS Pain Interference questionnaire.

    curl http://localhost:8081/fhir/Questionnaire/promis-pain-interference-6a-demo | jq
    Observe: The questionnaire structure, items, and answer options.

  • Task 1.2: Expand a ValueSet

    Test terminology service integration.

    curl "http://localhost:3000/loinc/AnswerList/LL358-3" | jq

  • Task 1.3: Submit a QuestionnaireResponse

    Create a sample response on Site A.

    curl -X POST http://localhost:8081/fhir/QuestionnaireResponse \
  -H "Content-Type: application/fhir+json" \
  -d '{
    "resourceType": "QuestionnaireResponse",
    "questionnaire": "http://loinc.org/Questionnaire/62629-8",
    "status": "completed",
    "authored": "2024-11-27T12:00:00Z",
    "item": [{
      "linkId": "61758-9",
      "answer": [{
        "valueCoding": {
          "system": "http://loinc.org",
          "code": "LA13914-9",
          "display": "Quite a bit"
        }
      }]
    }]
  }' | jq

  • Task 1.4: Transfer to Site B

    Retrieve from Site A and post to Site B.

    # Get from Site A
RESPONSE=$(curl -s http://localhost:8081/fhir/QuestionnaireResponse/demo-response-1)

# Post to Site B
echo $RESPONSE | curl -X POST http://localhost:8082/fhir/QuestionnaireResponse \
  -H "Content-Type: application/fhir+json" -d @-
Key Learning: Basic FHIR operations work. The challenge is in the details: versioning, scoring, translations.

2. Version Mismatch Detection Issue

Discover how scoring algorithm versions can silently cause incorrect results.

  • Task 2.1: Compare PROMIS v1.0 vs v2.0 ordinals

    Notice how the same answer has different ordinal values.

    # Version 1.0 ordinals
curl "http://localhost:3000/loinc/AnswerList/LL358-3?version=1.0" | jq '.concept[] | {display, ordinal}'

# Version 2.0 ordinals (REVERSED!)
curl "http://localhost:3000/loinc/AnswerList/LL358-3?version=2.0" | jq '.concept[] | {display, ordinal}'

  • Task 2.2: Run the version mismatch scenario

    Use the Demo UI or API to see the impact.

    curl http://localhost:8000/api/scenarios/version-mismatch | jq

  • Task 2.3: Calculate the score difference

    If a patient selects "Not at all" for all 6 items:

    • v1.0 scoring: Raw score = ?
    • v2.0 scoring: Raw score = ?
    • What's the clinical implication?

  • Task 2.4: Find version information in resources

    Where would you store/find the scoring algorithm version?

    # Check the questionnaire metadata
curl http://localhost:8081/fhir/Questionnaire/promis-pain-interference-6a-demo | jq '.version, .extension'
Key Learning: Same response + different versions = different scores. There's no standard FHIR extension for scoring algorithm version. This is a patient safety risk.

3. Missing Translation Handling Gap

Explore multilingual support gaps and fallback behavior.

  • Task 3.1: Get German translation

    German translations are complete.

    curl "http://localhost:3000/loinc/AnswerList/LL358-3?displayLanguage=de" | jq '.concept[] | {code, display}'

  • Task 3.2: Get Turkish translation

    Turkish translations are partially missing.

    curl "http://localhost:3000/loinc/AnswerList/LL358-3?displayLanguage=tr" | jq '.concept[] | {code, display}'
    Observe: Some displays show [TR: Not available]

  • Task 3.3: Run the missing translation scenario

    curl http://localhost:8000/api/scenarios/missing-translation | jq

  • Task 3.4: Design a fallback strategy

    What should happen when a translation is missing?

    • Show English fallback?
    • Show error/warning?
    • Block the questionnaire?
    • How should this be indicated in FHIR?
Key Learning: There's no standardized fallback mechanism for missing translations. Each product must implement its own logic.

  • Task 4.2: Compare with standard LOINC FHIR API

    What operations are available in standard LOINC FHIR?

    # Standard LOINC FHIR only provides CodeSystem lookup
# No panel structure, no Questionnaire generation

  • Task 4.3: Get answer list with ordinal values

    curl "http://localhost:3000/loinc/AnswerList/LL358-9?include=ordinals" | jq

  • Task 4.4: Identify needed API enhancements

    What APIs should LOINC FHIR provide that it doesn't?

    • Panel to Questionnaire conversion?
    • Ordinal values in expansions?

Key Learning: Standard LOINC FHIR APIs lack panel structure operations. Enhanced terminology services can fill the gap, but should this be standardized?

5. Discussion & Next Steps

Technical Challenges Identified

  • Version Management

    No standard extension for scoring algorithm version. Detection requires out-of-band knowledge.

  • Terminology Gaps

    LOINC panel structure not in FHIR API. Ordinal values inconsistently included.

  • Translation Handling

    No standard fallback mechanism. Completeness not indicated in metadata.

  • Product Integration

    EHR systems don't render FHIR Questionnaires natively.

Potential Pilot Projects

  • Enhanced Terminology Proxy

    LOINC panel APIs, ordinal values, multi-language support.

  • PROMIS Scoring Service

    Centralized IRT calculations with version management.

  • Version Compatibility Checker

    Detect mismatches, recommend migrations.
